\section*{Status of This Memo}
\vspace{-4pt}
{\small This document specifies an Internet standards-track protocol for the Internet
community, and requests discussion and suggestions for improvement. Distribution of this
memo is unlimited. It is subject to the terms of BCP 78 and the Trust Legal Provisions.}

\begin{lstlisting}
Abstract
   BAFP/2 is a low-latency, bidirectional framing protocol for
   multiplexed streams over a single transport connection. It adds
   adaptive flow control and per-stream prioritization to BAFP/1
   while preserving wire compatibility for the base header.
\end{lstlisting}

\section{Introduction}
This document defines version 2 of the Bidirectional Adaptive Frame Protocol (BAFP/2).
BAFP/2 multiplexes independent streams over one transport connection, providing per-stream
flow control and prioritization with minimal head-of-line blocking.

\subsection{Requirements Language}
The key words \bcp{MUST}, \bcp{MUST NOT}, \bcp{REQUIRED}, \bcp{SHALL}, \bcp{SHOULD}, and
\bcp{MAY} in this document are to be interpreted as described in BCP\,14 [RFC2119] [RFC8174]
when, and only when, they appear in all capitals.

\section{Frame Format}
Every BAFP/2 frame consists of a 12-byte base header, zero or more variable-length
extension entries, and a payload. All multi-byte integers are in network byte order.

\begin{table}[ht]\centering
\rowcolors{2}{light}{white}
\begin{tabularx}{\linewidth}{@{}l c X@{}}
\rowcolor{ink}
{\color{white}\textbf{Field}} & {\color{white}\textbf{Bits}} & {\color{white}\textbf{Description}}\\
Version   & 4  & Protocol version; \bcp{MUST} be set to \texttt{0x2}.\\
Type      & 4  & Frame type (DATA, HEADERS, SETTINGS, PING, GOAWAY).\\
Flags     & 8  & Type-specific boolean flags.\\
Stream ID & 32 & Unsigned stream identifier; 0 is reserved for connection control.\\
Sequence  & 16 & Monotonic per-stream sequence number.\\
Length    & 32 & Length of the payload in octets.\\
\end{tabularx}
\caption{BAFP/2 base header fields (12 octets total).}
\end{table}

\section{Flow Control}
BAFP/2 uses a credit-based scheme. A receiver advertises an initial window via the
\texttt{SETTINGS} frame. A sender \bcp{MUST NOT} send more payload octets on a stream than
the receiver has granted. Credits are replenished with \texttt{WINDOW\_UPDATE} frames.

\subsection{Prioritization}
Each stream carries an 8-bit weight. Endpoints \bcp{SHOULD} allocate bandwidth in
proportion to weight among streams that are ready to send.

\section{Error Handling}
An endpoint that detects a protocol violation \bcp{MUST} terminate the connection with a
\texttt{GOAWAY} frame carrying the last processed stream ID and an error code.

\begin{tabularx}{\linewidth}{@{}l l X@{}}
\toprule
\textbf{Code} & \textbf{Name} & \textbf{Meaning}\\
\midrule
0x0 & NO\_ERROR        & Graceful shutdown.\\
0x1 & PROTOCOL\_ERROR  & Malformed frame or state violation.\\
0x3 & FLOW\_CONTROL    & Peer exceeded advertised window.\\

\end{table}

\section{Security Considerations}
BAFP/2 \bcp{MUST} be run over a transport providing confidentiality and integrity (e.g.\
TLS 1.3). Implementations \bcp{SHOULD} bound the number of concurrent streams to mitigate
resource-exhaustion attacks.

\section{IANA Considerations}
IANA is requested to create a ``BAFP/2 Frame Types'' registry and a ``BAFP/2 Error Codes''
registry, both with a Specification Required policy.
