% ── Executive Summary ────────────────────────────────────────
\section{Executive Summary}

On \textbf{2024-11-14} at 03:17 UTC, a misconfigured automated maintenance window
triggered an unplanned failover of the primary Vertex Aurora PostgreSQL cluster
(\texttt{payments-db-us-east-1}) serving the Payment Processing API. The failover
propagated a cascade failure into the API Gateway layer due to hardcoded connection
pool exhaustion logic. Approximately \textbf{47\,000 end users} were unable to
complete checkout flows for \textbf{2 hours 43 minutes}. Total estimated revenue
impact is \textbf{\$183\,000} with an additional \$24\,000 in cloud cost overrun
from emergency scaling actions. This document captures the full timeline, root
cause analysis, blast radius, and the set of action items committed to by the
team to prevent recurrence.

\begin{callout}[title={\faIcon{lightbulb}\hspace{4pt}Key Finding}]{accent}
The failure was \emph{not} caused by the database failover itself (which completed
successfully in 32\,s) but by the API service's connection pool refusing to
re-establish connections after the new primary endpoint's DNS propagated, due to a
missing \texttt{read\_after\_write} retry wrapper introduced in \texttt{v4.7.2}.
\end{callout}

\subsection{Contributing Factors}

\begin{enumerate}[leftmargin=*, itemsep=2pt, label=\textcolor{accent}{\bfseries\arabic*.}]
  \item \textbf{Misconfigured Maintenance Window}: The automation runbook targeted
    \texttt{env=production} instead of \texttt{env=staging} due to a copy-paste error
    in the Terraform variable file committed 3 days prior.

  \item \textbf{Missing Canary Test}: Deployment pipeline for \texttt{v4.7.2} had canary
    gates disabled for ``hotfix track'' releases, bypassing the integration suite that
    would have caught the missing retry wrapper.

  \item \textbf{Insufficient Observability}: No dedicated metric for ``connection pool
    exhaustion'' existed; the signal appeared only as generic latency, delaying diagnosis
    by $\approx$14 minutes.

  \item \textbf{On-call Runbook Gap}: The DB failover runbook did not include a step to
    verify API-layer reconnection behaviour post-failover, leaving the on-call engineer
    without a clear path to diagnosis.
\end{enumerate}

\subsection{What Went Well}

\begin{enumerate}[leftmargin=*, itemsep=2pt, label=\textcolor{statusdone}{\bfseries\arabic*.}]
  \item PagerDuty alert fired within \textbf{8 minutes} of first errors, meeting the
    \textless{}10-minute SLO for P99 threshold breach.
  \item War room assembled and effective within 10 minutes of acknowledgement.
  \item Aurora failover itself completed in \textbf{32 seconds} --- well within the
    60-second RTO target for the database tier.
  \item Feature-flag rollback of the pool config was executed in under 3 minutes once
    the root cause was identified.
  \item Read replica traffic remained fully available; browse and search features were
    unaffected throughout the incident.
\end{enumerate}

% ── Code Evidence ────────────────────────────────────────────
\subsection{Code Evidence --- Missing Retry Wrapper}

The following diff shows the \texttt{v4.7.2} change that inadvertently removed
the Aurora reconnection wrapper:

\begin{lstlisting}[language=Python, caption={\texttt{src/db/pool.py} --- diff v4.7.1 to v4.7.2}]
- async def get_connection(self) -> Connection:
-     for attempt in range(self.max_retries):
-         try:
-             conn = await self._pool.acquire(timeout=2.0)
-             await conn.execute("SELECT 1")  # probe after failover
-             return conn
-         except (PoolExhaustedError, ConnectionRefusedError):
-             await asyncio.sleep(0.5 * (2 ** attempt))
-     raise DatabaseUnavailableError("Pool exhausted after retries")
+
+ async def get_connection(self) -> Connection:
+     return await self._pool.acquire()  # no retry, no probe
\end{lstlisting}

The Aurora cluster endpoint (\texttt{cluster.proxy-xxx.rds.amazonaws.com})
updated DNS within 32 seconds of failover, but in-flight pool connections
cached the old primary IP. Without the probe, callers received stale
connections that immediately failed, flooding the pool.

% ── Lessons Learned ──────────────────────────────────────────
\section{Lessons Learned}

\begin{infobox}
\setlength{\parskip}{0pt}
\begin{tabularx}{\linewidth}{@{}>{\bfseries\small\color{accent}}l@{\hspace{6mm}}>{\small\raggedright\arraybackslash}X@{}}
Lesson 1 & \textbf{Failover $\neq$ Transparent}: A database failover completing in 32\,s
            does not guarantee application-tier transparency. Each service must be tested
            independently against failover events.\\[4pt]
Lesson 2 & \textbf{Hotfix Shortcuts Backfire}: Disabling canary gates for speed during
            hotfix releases introduces exactly the kind of regression that requires a longer
            incident response. The tradeoff is almost never worth it.\\[4pt]
Lesson 3 & \textbf{Observability Gaps Kill Diagnosis Speed}: 14 minutes were lost because
            there was no metric distinguishing ``connection pool exhaustion'' from generic
            latency. Instrument what you need to triage, not just what you think will
            fail.\\[4pt]
Lesson 4 & \textbf{Runbooks Age Poorly}: The DB failover runbook was 18 months old and
            did not reflect the current API architecture. All Tier-1 runbooks should be
            reviewed quarterly and tested against actual chaos drills.\\
\end{tabularx}

\end{infobox}
