% ============================================================
\section{Context \& Scope}
% ============================================================

\begin{contextbox}
\textbf{Problem.}\ The current single-ledger Postgres monolith cannot sustain
the projected 40$\times$ write throughput required for the 2026 Stripe-level
merchant-acquiring rollout. P99 write latency breached 800\,ms during the Q3
Black Friday stress test. This document proposes replacing the hot-path write
layer with a partitioned, event-sourced ledger backed by Apache Kafka and a
TiDB-distributed storage tier.

\vspace{4pt}
\textbf{Scope.}\ Covers the \icode{ledger-write-service}, \icode{ledger-query-service},
Kafka topic schema, TiDB shard strategy, and the zero-downtime migration plan.
Out of scope: fraud-scoring pipeline, merchant portal UI, settlement batch jobs.
\end{contextbox}

\vspace{6pt}

\textbf{Background.}\ Stripe's public post-mortem (2024) attributes 68\,\% of
payment-latency incidents to single-region database serialisation under high
concurrency. Our current architecture mirrors this fragility:\ one
\icode{payments\_ledger} table with 3.2\,B rows, 14 secondary indexes, and
no horizontal sharding. The re-architecture targets:

\begin{itemize}
  \item Sub-50\,ms P99 write latency at 50\,k TPS sustained.
  \item Idempotent, exactly-once debit/credit semantics via outbox pattern.
  \item GDPR-compliant PII isolation per merchant shard.
  \item RPO $\leq$ 1\,s, RTO $\leq$ 30\,s in a region failover.
\end{itemize}

\vspace{4pt}

\begin{warnbox}[Decision Required by 2026-01-20]
TiDB vs.\ CockroachDB vendor choice is not yet final.
This doc assumes TiDB; \S11 covers CockroachDB as the primary alternative.
The infra cost delta is \$180k/yr. Pending CFO sign-off.
\end{warnbox}

% ============================================================
\section{Goals \& Non-Goals}
% ============================================================

\subsection{Goals}
\begin{enumerate}
  \item \textbf{Throughput.}\ Handle 50\,k TPS peak write load with P99 $\leq$ 50\,ms.
  \item \textbf{Durability.}\ Zero data loss on single-node failure; 3-way
        replication across AZs.
  \item \textbf{Idempotency.}\ Duplicate payment events must be deduplicated
        within a 24\,h window using \icode{idempotency\_key}.
  \item \textbf{Zero-downtime migration.}\ Dual-write bridge with read-traffic
        cut-over under feature flag; full cut-over in $\leq$ 4\,h maintenance window.
  \item \textbf{Observability.}\ Every write emits a structured OpenTelemetry span;
        Datadog dashboard auto-provisioned via Terraform.
\end{enumerate}

\subsection{Non-Goals}
\begin{itemize}
  \item Real-time fraud inference — remains in the existing ML scoring service.
  \item Re-architecture of the settlement or reconciliation batch tier.
  \item Support for non-USD currency ledgers (planned for Q3 2026, separate RFC).
\end{itemize}

\subsection{Key Design Decisions}

\begin{tabularx}{\linewidth}{@{}K@{\hspace{8mm}}V@{}}
\toprule
\textbf{Concern} & \textbf{Decision} \\
\midrule
Write path & Outbox pattern: write to \icode{ledger\_outbox} in same TiDB txn, Kafka relay picks up async. Guarantees atomicity without distributed 2PC. \\
Partition key & \icode{merchant\_id \% 128} — keeps all events for a merchant on one partition (ordering guarantee). \\
Idempotency & \icode{idempotency\_key} stored in Redis (TTL 24\,h); checked before outbox insert. \\
Schema evolution & Protobuf v3 with mandatory \icode{schema\_version} field; backward-compat enforced by schema registry. \\
Fan-out reads & CQRS: query service consumes Kafka and projects into Elasticsearch for merchant-facing queries; TiDB for internal accounting. \\
\bottomrule
\end{tabularx}

\vspace{6pt}

\subsection{Outbox Relay}

The relay is a single Go goroutine per TiDB shard using \icode{SELECT FOR UPDATE
SKIP LOCKED} on the \icode{ledger\_outbox} table. Batch size: 500 rows, flush
interval: 50\,ms. On Kafka producer ACK the rows are deleted. Relay lag is
exposed via \icode{ledger\_outbox\_depth\_gauge} (alert if $>$ 10\,k).

\subsection{Protobuf Schema — \texttt{ChargeRequest}}

\begin{lstlisting}[language=proto]
syntax = "proto3";
package payments.ledger.v2;

message ChargeRequest {
  string  idempotency_key  = 1;  // UUID v4, required
  string  merchant_id      = 2;  // e.g. "mid_4xR9a..."
  int64   amount_cents      = 3;  // positive, USD cents
  string  currency          = 4;  // ISO 4217 ("USD")
  string  description       = 5;  // max 255 chars
  map<string, string> metadata = 6;
  int32   schema_version    = 7;  // must == 2
}

message ChargeResponse {
  string  charge_id     = 1;  // "chg_..." globally unique
  string  status        = 2;  // PENDING | COMPLETED | FAILED
  int64   balance_cents  = 3;
  int64   created_at_us  = 4;  // Unix microseconds
}

service LedgerService {
  rpc Charge  (ChargeRequest)  returns (ChargeResponse);
  rpc Refund  (RefundRequest)  returns (RefundResponse);
  rpc Balance (BalanceRequest) returns (BalanceResponse);
}
\end{lstlisting}

\subsection{TiDB Schema}

\begin{lstlisting}[language=SQL]
CREATE TABLE ledger_events (
    event_id        CHAR(36)      NOT NULL,
    merchant_id     VARCHAR(64)   NOT NULL,
    amount_cents    BIGINT        NOT NULL,
    currency        CHAR(3)       NOT NULL DEFAULT 'USD',
    event_type      ENUM('CHARGE','REFUND','PAYOUT') NOT NULL,
    idempotency_key CHAR(36)      NOT NULL,
    created_at      DATETIME(6)   NOT NULL,
    metadata        JSON,
    PRIMARY KEY (merchant_id, created_at, event_id)  -- composite PK = shard key
) PARTITION BY HASH(merchant_id) PARTITIONS 128;

CREATE TABLE ledger_outbox (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    event_id    CHAR(36)    NOT NULL,
    payload     BLOB        NOT NULL,  -- serialised proto
    created_at  DATETIME(6) NOT NULL,
    published   TINYINT(1)  NOT NULL DEFAULT 0,
    INDEX idx_unpublished (published, created_at)
);
\end{lstlisting}

\section{Migration Plan}
% ============================================================

\begin{warnbox}[Migration Risk: HIGH]
The existing ledger table has 3.2\,B rows and 14 live consumer services.
A phased cut-over with automated rollback is mandatory.
\end{warnbox}

\subsection{Phases}

\begin{tabularx}{\linewidth}{@{}>{\bfseries\color{accent}}l@{\hspace{6mm}}l@{\hspace{4mm}}V@{}}
\toprule
Phase & Window & Description \\
\midrule
0 & Week 0 & Provision TiDB cluster; schema DDL; seed with last 90d of history via Spark job. Shadow-write mode (writes go to both old DB and TiDB, reads from old). \\
1 & Week 1--2 & Enable dual-write via feature flag \icode{ledger\_dual\_write}. Monitor TiDB lag vs.\ Postgres baseline. Alert on $>$ 0.1\,\% divergence. \\
2 & Week 3 & Shift 10\,\% of read traffic to TiDB (canary). Validate P99 latency and correctness via checksum comparison job. \\
3 & Week 4 & Ramp reads to 100\,\% TiDB. Keep Postgres in standby for 48\,h rollback window. \\
4 & Week 5 & Disable Postgres writes. Decommission old DB after 30-day hold for audit. \\
\bottomrule
\end{tabularx}

\vspace{6pt}

\subsection{Rollback Procedure}

In Phase 1--3, flip \icode{ledger\_dual\_write=false} in LaunchDarkly. Traffic
immediately reverts to Postgres. Estimated rollback time: $<$ 30\,s (flag
propagation SLA). Phase 4 rollback requires point-in-time recovery from Postgres
daily snapshot — RTO $\approx$ 4\,h.

\begin{infobox}[Automated Divergence Check]
  A nightly reconciliation job (\icode{ledger-recon-job}) runs a row-count and
  checksum comparison between Postgres and TiDB for the previous 24\,h window.
  Any delta $>$ 0.01\,\% pages the on-call via PagerDuty and halts Phase advancement.
\end{infobox}

% ============================================================
\section{Open Questions}
% ============================================================

\begin{enumerate}
  \item \textbf{[Infra]} Should the Kafka relay run as a sidecar to \icode{ledger-write-service}
        or as a standalone \icode{ledger-relay-service}? Sidecar reduces network hops;
        standalone allows independent scaling. Decision needed by 2026-01-25.

  \item \textbf{[Product]} Do we need sub-second balance reads during Phase 1 dual-write,
        or is eventual consistency (100\,ms lag from Kafka) acceptable for merchant dashboards?

  \item \textbf{[Legal]} GDPR merchant data deletion — is a 30-day TiDB shard tombstone
        compliant, or must deletion be synchronous?

  \item \textbf{[Security]} Should \icode{idempotency\_key} be stored in plaintext in Redis,
        or HMAC'd with merchant secret? K.\ Narayanan to advise by 2026-01-18.
\end{enumerate}

\section{Alternatives Considered}
% ============================================================

\begin{tabularx}{\linewidth}{@{}K@{\hspace{6mm}}V@{}}
\toprule
Alternative & Verdict \\
\midrule
CockroachDB & Comparable feature set, 22\,\% higher p99 in our benchmark at 50\,k TPS (see attached Notion doc). Cockroach charges per vCPU; \$180k/yr delta vs TiDB. Keep as fallback if TiDB contract fails. \\
Postgres + Citus & Evaluated Q2 2025. Citus shard rebalancing caused 4-min read unavailability in staging under 30\,k TPS. Disqualified. \\
PlanetScale (Vitess) & No multi-statement transactions — incompatible with outbox pattern. Disqualified. \\
Event sourcing on DynamoDB & Single-region without Global Tables; 10$\times$ cost at our volume. Disqualified. \\
