\chapter{Introduction}

\section{Scope}
This technical report documents the engineering development of the Advanced Flight
Control System (AFCS) Upgrade for the \aircraft\ aircraft. The AFCS replaces the
legacy analog flight control computer (FCC) with a modern digital triplex-redundant
architecture, providing enhanced stability augmentation, improved fault tolerance, and
full-authority fly-by-wire capability for selected flight regimes.

The scope encompasses:
\begin{itemize}
  \item System architecture definition and hardware/software partitioning.
  \item Functional requirements traceability from system-level specifications to
        hardware and software components.
  \item Verification planning aligned with MIL-STD-882E System Safety requirements.
  \item CDRL compliance mapping for all deliverable data items.
\end{itemize}

\section{System Overview}
The AFCS upgrade consists of three primary subsystems:

\begin{enumerate}[label=\textbf{\arabic*.}, leftmargin=*]
  \item \textbf{Flight Control Computer (FCC).} Triplex digital processor with
        dissimilar-channel voting logic. Each channel is a PowerPC~8640D running
        a DO-178C Level~A certified real-time executive.
  \item \textbf{Sensor Suite.} Dual-redundant GNSS-aided INS (LN-260), air-data
        computer (ADC-3000), and rate-gyro assembly providing $p,q,r$ body rates.
  \item \textbf{Actuation System.} Electro-hydrostatic actuators (EHAs) on elevator,
        aileron, and rudder surfaces, each with local loop-closure at 800~Hz.
\end{enumerate}

Redundancy management uses a median-select voting scheme with channel health
monitoring. Any single channel failure is transparent to the pilot; a second
failure degrades gracefully to direct-mode control.

  \item \docref{MIL-STD-882E} — Department of Defense Standard Practice: System Safety.

\chapter{System Architecture}

\section{Functional Architecture}
The AFCS functional architecture is organized into six processing layers,
as illustrated in Figure~\ref{fig:arch}. Sensor fusion occurs in the
\textit{Estimation} layer via a 15-state extended Kalman filter (EKF) outputting
aircraft state estimates at 200~Hz. The \textit{Guidance} layer computes
commanded trajectories from the flight management system. The
\textit{Control Allocation} layer maps generalized force/moment commands
to individual surface deflections using a real-time quadratic-programming solver.

\section{Avionics Bus Interface}
All inter-LRU communication uses dual-redundant MIL-STD-1553B buses
operating at 1~Mbit/s. Bus~A and Bus~B provide hot-standby redundancy;
the bus controller (BC) resides on FCC Channel~A with automatic failover
to Channel~B on loss of BC heartbeat. Each remote terminal (RT) responds
within the mandated 12~$\mu$s inter-message gap.

Figure~\ref{fig:milspec} shows the bus topology with the FCC as BC and
sensor/actuator units as RTs.

Each electro-hydrostatic actuator incorporates a local servo-loop closed
at 800~Hz with a proportional-integral (PI) current regulator. The command
signal from the FCC is a $\pm 10$~V analog reference representing desired
surface position. Figure~\ref{fig:eha} shows the drive-electronics interface.

\chapter{Requirements and Traceability}

\section{System-Level Requirements}
The following system-level functional requirements are derived from the
Performance Specification~\docref{AFCS-PS-001} and the System Specification~\docref{AFCS-SS-001}.

\begin{enumerate}[label=\req{\arabic*}, leftmargin=*, itemsep=6pt]
  \item \label{req:1}
    The AFCS shall provide three-axis stability augmentation with
    $\pm 0.05^\circ$ attitude hold accuracy in steady level flight.

  \item \label{req:2}
    The system shall detect and isolate any single-point failure within
    50~ms and reconfigure to an operable state without pilot intervention.

  \item \label{req:3}
    Control surface deflection bandwidth shall exceed 12~Hz at
    $-3$~dB for all primary surfaces (elevator, aileron, rudder).

  \item \label{req:4}
    The FCC shall execute one complete control-law cycle
    (sensor read $\rightarrow$ estimation $\rightarrow$ control
    $\rightarrow$ command output) within 5~ms (200~Hz update rate).

  \item \label{req:5}
    The AFCS shall maintain controlled flight through any single
    hydraulic or electrical power failure without exceeding
    $\pm 2\,g$ transient load factor.

  \item \label{req:6}
    Bus loading on each MIL-STD-1553B channel shall not exceed
    55\% of available bandwidth under worst-case traffic conditions.

  \item \label{req:7}
    The system MTBF shall exceed 5,000 flight hours as computed
    per MIL-HDBK-217F, Notice~2, Ground Benign environment.

  \item \label{req:8}
    Software shall be developed to DO-178C Design Assurance Level~A
    with full MC/DC coverage.
\end{enumerate}

\chapter{Verification and Validation}

\section{Verification Approach}
The AFCS verification program employs a four-tier strategy aligned with the
systems engineering V-model:

\begin{enumerate}[label=\textbf{Tier \arabic*:}, leftmargin=*]
  \item \textbf{Component-Level Bench Testing.} Each LRU is acceptance-tested
    against its procurement specification. The FCC undergoes burn-in at
    $+71^\circ$C for 168~hours per MIL-STD-810H Method~501.7.
  \item \textbf{Hardware-in-the-Loop (HIL) Simulation.} The integrated AFCS
    is exercised against a 6-DOF nonlinear airframe model in real time.
    A total of 427 test cases cover normal, degraded, and emergency operating
    modes.
  \item \textbf{Iron Bird Ground Testing.} A full-scale actuation test rig
    validates hinge-moment loading and flutter margins up to 1.2~$V_D$ (design
    dive speed).
  \item \textbf{Flight Test.} The upgraded AFCS will be flown on a dedicated
    F-15E test aircraft (S/N~86-0183) over 42 sorties totaling approximately
    65 flight hours. Test points are structured per
    MIL-HDBK-516C Paragraph~4.2.3.
\end{enumerate}
